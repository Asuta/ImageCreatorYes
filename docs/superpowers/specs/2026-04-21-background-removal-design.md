# 本地抠图集成设计

## 目标

为现有 Qwen 生图网页增加“生成后抠成透明 PNG”的能力。抠图能力不直接嵌入当前 Node 项目，而是通过配置本地 `BrainDeadBackgroundRemover` 可执行文件路径，由当前服务端在需要时调用其自动模式完成抠图。

## 背景与约束

- 现有项目已经具备图片生成能力，结果以图片 URL 的形式在前端卡片中展示。
- 用户选择的集成方式是“本地独立 EXE/CLI，由当前 Node 后端调用”。
- 第一版只做“对生成结果二次处理”，不做独立上传图片抠图。
- 第一版只使用 `BrainDeadBackgroundRemover` 的自动模式，不暴露复杂模型选项。
- 运行环境为 Windows，本地已有或将手动下载 EXE。

## 方案选择

### 采用方案

通过 `.env` 配置 `BrainDeadBGRemover.exe` 路径；当前 Node 服务新增抠图接口，在收到指定图片 URL 后：

1. 下载原图到临时目录
2. 调用本地 EXE 自动模式
3. 读取输出透明 PNG
4. 将输出文件保存到项目可访问目录
5. 返回抠图结果地址给前端

### 不采用的方案

- 将 BrainDeadBackgroundRemover 的模型/推理逻辑直接嵌入 Node 项目：复杂度高，升级困难。
- 浏览器直接调用本地 EXE：不符合浏览器安全模型，且维护困难。
- 直接接远程抠图 API：当前用户已经明确偏好本地模型。

## 配置设计

新增以下环境变量：

- `BRAINDEAD_BG_REMOVER_PATH`：本地 `BrainDeadBGRemover.exe` 绝对路径

可选增强（第一版可不启用）：

- `BRAINDEAD_BG_OUTPUT_DIR`：抠图输出目录，默认项目内 `generated/cutouts`

若未配置 EXE 路径，后端返回清晰错误信息，引导用户补充配置。

## 后端设计

### 新接口

- `POST /api/cutout`

请求体：

```json
{
  "imageUrl": "https://..."
}
```

返回体：

```json
{
  "imageUrl": "/generated/cutouts/xxx.png"
}
```

### 处理流程

1. 校验 `imageUrl`
2. 校验 `BRAINDEAD_BG_REMOVER_PATH` 是否存在
3. 下载远程图片到临时输入文件
4. 调用 `BrainDeadBGRemover.exe` 自动模式完成处理
5. 将输出 PNG 写入静态可访问目录
6. 删除中间临时文件
7. 返回结果 URL

### 文件组织

- 临时输入目录：项目内临时目录，例如 `tmp/cutout`
- 输出目录：`generated/cutouts`

Node 服务需要为输出目录提供静态访问能力。

## 前端设计

- 在每张生成结果卡片下新增按钮：`抠成 PNG`
- 点击后按钮进入处理中状态
- 完成后：
  - 卡片中增加透明 PNG 预览或替换为抠图结果
  - 提供“打开 PNG”或“下载 PNG”链接
- 抠图失败时，在状态区或卡片上显示错误信息

## 错误处理

- EXE 路径未配置：提示用户在 `.env` 中配置路径
- EXE 文件不存在：返回明确路径错误
- 下载原图失败：提示图片不可访问
- EXE 执行失败：返回 stderr 或简化后的失败原因
- 输出文件不存在：提示抠图处理未生成结果

## 安全与边界

- 仅允许处理当前服务端可下载的图片 URL
- 仅在后端调用本地 EXE，不在浏览器暴露本地路径
- 第一版不做任务队列，不做并发调度优化

## 测试策略

- 单元测试：
  - 验证配置缺失时报错
  - 验证抠图任务参数构造正确
- 手工验证：
  - 生成图片后点击“抠成 PNG”
  - 路径未配置时确认报错可读
  - 配置正确时确认返回透明背景 PNG

## 非目标

- 不实现任意本地图像上传抠图
- 不开放模型选择、阈值、边缘细化等高级参数
- 不做结果历史记录管理
