# 参考图上传设计

## 目标

为现有 Qwen 生图网页增加“本地参考图上传”能力。用户可以在表单中选择最多 3 张本地图片，提交后由前端读取为 Base64 Data URL，再由后端转发到阿里百炼兼容接口。未上传参考图时，继续保持现有纯文本生图流程。

## 背景与约束

- 现有项目已经支持 `qwen-image-2.0` 和 `qwen-image-2.0-pro` 的纯文生图。
- 参考图输入需遵循阿里百炼官方图像编辑/多模态输入格式：`input.messages[0].content` 中可包含若干 `{"image":"..."}` 项以及一个 `{"text":"..."}` 项。
- 参考图来源限定为用户本地选择的文件，不新增对象存储、数据库或第三方上传流程。
- 单次最多支持 3 张参考图，超出时应拒绝提交。

## 方案选择

### 采用方案

前端将本地文件读取为 Base64 Data URL，并与提示词一起提交给本地代理服务；后端根据是否存在参考图，组装兼容的 `content` 数组并转发到现有接口。

### 不采用的方案

- 先上传到 OSS 再传 URL：实现复杂度高，不符合当前最小可用目标。
- 仅支持用户填写公网图片 URL：不能满足“选择本地图片文件”的需求。

## 前端设计

- 在现有表单中新增 `input[type=file]`，允许多选图片文件。
- 限制可选文件类型为常见图片格式，限制数量为最多 3 张。
- 提交前将每张图片读取为 Data URL，并和其基础元信息一起放入请求体。
- 页面展示已选文件名和张数；若超过数量限制或读取失败，给出明确错误提示。
- 未选参考图时，不改变现有交互。

## 后端设计

- 扩展 `/api/generate` 请求体结构，允许接收 `referenceImages` 数组。
- 在 payload 构造逻辑中：
  - 无参考图：保持 `content: [{ text: prompt }]`
  - 有参考图：生成 `content: [{ image: dataUrl1 }, ..., { image: dataUrlN }, { text: prompt }]`
- 后端再次校验参考图数量上限，避免仅靠前端限制。
- 保持现有鉴权方式与接口地址不变。

## 数据流

1. 用户选择 0-3 张本地图片并填写提示词。
2. 前端将本地文件转为 Base64 Data URL。
3. 前端向 `/api/generate` 发送 JSON，请求中包含模型参数、提示词、可选参考图数组。
4. 后端构造百炼兼容 payload 并调用上游接口。
5. 后端提取返回图片 URL，前端沿用现有结果展示逻辑。

## 错误处理

- 前端：
  - 超过 3 张时阻止提交并提示。
  - 文件读取失败时提示具体失败信息。
- 后端：
  - 缺少提示词仍报错。
  - 参考图数量超过 3 张时报错。
  - 上游接口失败时继续透传错误信息。

## 测试策略

- 单元测试覆盖：
  - 无参考图时 payload 保持兼容。
  - 有 1-3 张参考图时 payload 中 `content` 顺序正确。
  - 超过 3 张参考图时抛出错误。
- 手工验证覆盖：
  - 仅文本生成。
  - 单张参考图生成。
  - 多张参考图生成。

## 非目标

- 不实现图片裁剪、压缩、拖拽排序。
- 不接入文件持久化存储。
- 不新增历史记录与任务队列。
